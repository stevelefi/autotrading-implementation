# Implementation Instructions

## Purpose
Production-standard workflow for implementing and shipping changes in this repository.

## 1. Baseline and Contract Discipline

1. `SPEC_VERSION.json` is authoritative for the spec baseline (current: `spec-v1.0.1-m0m1`).
2. Before coding, run spec sync + verify:
   ```bash
   python3 tools/spec_sync.py sync \
     --repo-url https://github.com/stevelefi/autotrading.git \
     --ref spec-v1.0.1-m0m1 \
     --dest specs/vendor \
     --version-file SPEC_VERSION.json

   python3 tools/spec_sync.py verify --dest specs/vendor --version-file SPEC_VERSION.json
   ```
3. Read required docs before writing code:
   - `specs/vendor/docs/contracts/**`
   - `specs/vendor/docs/source-of-truth/**`
   - `specs/vendor/docs/repo-charters/autotrading-implementation.md`
4. Runtime API/topic/proto/schema changes are **frozen during blitz**. Raise a spec-change request in the spec repo first.

## 2. Branch and PR Standard

1. Branch naming: `codex/<repo>-<issue>-<short-topic>`
2. Every PR must include:
   - Linked issue (`Closes #<id>` / `Fixes #<id>` / `Resolves #<id>`)
   - Pinned spec ref
   - Acceptance checklist
   - Evidence links (test results, drill logs, runbook updates)

## 3. Repository Hygiene

1. Never commit build outputs (`**/target/`, logs, temp files) â€” covered by `.gitignore`.
2. Never commit secrets (`.env`, `.env.local`, credential files).
3. Commit templates only (`.env.example`, `infra/local/.env.compose.example`).
4. `specs/vendor/` is gitignored â€” it is regenerated by spec sync, never hand-edited.

## 4. Required Local Validation Before Commit

```bash
# 1. Verify spec baseline
python3 tools/spec_sync.py verify --dest specs/vendor --version-file SPEC_VERSION.json

# 2. Unit + integration tests (all 12 Maven modules)
mvn -B test

# 3. Coverage gate (reliability-core path)
make test-coverage-core

# 4. Helm lint + template render
helm lint infra/helm/charts/trading-service
helm template trading-service infra/helm/charts/trading-service \
  -f infra/helm/charts/trading-service/values.yaml

# 5. Optional compose config check
docker compose --env-file infra/local/.env.compose.example \
  -f infra/local/docker-compose.yml config

# 6. Runtime smoke (required for any runtime-touching change)
make up && make smoke-local && make down
```

## 5. Reliability and Safety Expectations

### Idempotency
- Same key + same payload â†’ replay existing acceptance (`COMMAND_STATUS_DUPLICATE`)
- Same key + different payload â†’ conflict (`COMMAND_STATUS_REJECTED`)

### Outbox/Inbox
- Outbox row appended in **same transaction** as domain mutation
- `ConsumerDeduper` checks `consumer_inbox` before processing â€” duplicate consumer event is a no-op
- In-memory dedup state is **intentionally not kept** between restarts; `ConsumerDeduper` is the authoritative dedup layer

### Command Safety
- Retry with the same idempotency key cannot create a duplicate broker submit
- Missing first status within 60 seconds triggers `UNKNOWN_PENDING_RECON` state and `FROZEN` trading mode
- Signal persisted to DB **before** gRPC call to risk-service (crash-safe: signal is always recoverable)
- Kill-switch and trading-mode restored from `system_controls` table on `monitoring-api` startup

### gRPC Error Handling
- All gRPC service methods have `try-catch` with `INVALID_ARGUMENT` or `INTERNAL` status mapping
- `MDC.clear()` in `finally` blocks on all gRPC service methods â€” prevents MDC key leakage across thread pool requests
- gRPC server lifecycle: `awaitTermination(5s)` before `shutdownNow()` to drain in-flight RPCs gracefully

### Thread Safety
- `alertEvents` and similar shared lists use `CopyOnWriteArrayList`
- ConcurrentHashMap position updates use `compute()` for atomic read-modify-write
- gRPC `server` fields are `volatile`
